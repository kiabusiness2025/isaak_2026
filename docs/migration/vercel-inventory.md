# Isaak Retro — Auditoría Fase 0: inventario de proyectos Vercel (`vercel-inventory.md`)

## Resumen

Ocho `apps/*` tienen `vercel.json` propio (`isaak`, `app`, `admin`, `landing`, `holded`, `client`, `api`, `holded-mcp`) — cada uno es un proyecto Vercel independiente (monorepo-style, `installCommand: cd ../.. && pnpm install --frozen-lockfile`). `apps/mobile` (Flutter), `apps/isaak_retro_vibe` y `apps/isaak-video-generator` no tienen `vercel.json` (no desplegados como proyecto Vercel — este último usa Vercel Blob solo para storage, ver tabla). El `vercel.json` de la raíz del repo (`outputDirectory: "public"`) parece un artefacto sin uso — no existe carpeta `public/` en la raíz y no hay `.vercel/` local en ningún sitio del repo.

## Inconsistencia de dominio a resolver

El README raíz dice que `apps/app` → `isaak.chat`, pero varios `.env.example` y documentación (`apps/holded`, `apps/admin`, `apps/holded-mcp`) todavía referencian `apps/app` como `app.verifactu.business`. `docs/engineering/DOMAIN_MIGRATION_PLAN_2026-06-27.md` documenta esta migración de dominio como **en curso**, no completada en toda la documentación.

**Relevancia para `isaak_retro`:** confirma que fijar el esquema de dominios (`isaak.es` + `app.isaak.es` + `admin.isaak.es`, según decisión ya tomada) desde el día 1 del repo nuevo evita repetir este tipo de inconsistencia arrastrada.

## Turbo remote cache

`turbo.json` no tiene `remoteCache` configurado explícitamente — solo declara el allowlist `globalEnv` (~120 vars) y los `outputs` por tarea (`.next/**`, `dist/**` de cada app).
